# dev-task

開発タスクを軽量かつ高品質に実装する Skill。

入力は 2 形式:

- **Linear チケット ID** (`[A-Z]+-\d+` 形式) → Linear MCP でチケット本体・コメント・関連 diff・親プロジェクトを取得
- **自然言語の説明** → メッセージから意図と受け入れ条件を抽出

対象スタック: バックエンド Kotlin、フロントエンド React TypeScript、必要に応じて protobuf と Node.js。

## ワークフロー

1. **入力解決** — Linear チケットまたは自然言語から作業仕様(課題 + 受け入れ条件)を構築
2. **コンテキスト収集** — 関連ファイル・既存パターン・類似実装を探索
3. **トリアージ** — 直接実装するか、プラン承認後に実装するかを判定(後述の*プラン提示シグナル*)
4. **実装** — *実装ルール*に従う。UI 系のタスクは `frontend-fidelity` Skill を起動して委譲(後述の*UI 委譲プロトコル*)
5. **検証** — プロジェクトが対応する型/ビルド/テスト/lint を実行(委譲した場合も dev-task が必ず実行)

## フェーズ 1: 入力解決

メッセージに Linear チケット ID が含まれる場合:

- Linear MCP でチケット本体とコメントを取得。関連 diff、添付、参照されている親プロジェクトの説明も読む。
- 抽出する情報: 課題、受け入れ条件、言及されているファイル/コンポーネント、関連チケット。

自然言語の場合:

- 自己確認のためタスクを 1 行で言い直す。自明な場合は確認待ちで止まらない。
- 分類する: バグ修正 / 機能追加 / リファクタ / 設定変更。

受け入れ条件が欠落または曖昧な場合、**質問返しせず仮定を明示的に記録する**。非自明な変更であればプラン提示時にユーザーに見せる。

## フェーズ 2: コンテキスト収集

コードに触れる前に:

- 影響範囲のレイヤーを特定: バックエンド Kotlin / フロント React-TS / proto / Node.js (複数のことも多い)
- 該当箇所を特定: ファイル glob、タスクのキーワードで grep、import / 使用箇所をたどる
- **類似する既存実装を最低 1 つ見つける。** これがフェーズ 4 で模倣すべきパターンになる
- 触れる言語領域ごとに、対応する `references/` を読む

**フロントエンドの UI 実装作業**(React コンポーネント、ページ、Figma 起点の変更)はフェーズ 4 を `frontend-fidelity` に委譲する。`dev-task` は入力解決・コンテキスト収集・プラン・検証を引き続き担当。委譲の具体的な手順とフォーマットは下記「UI 委譲プロトコル」を参照。

## フェーズ 3: トリアージ — 直接実装 or プラン提示

**直接実装してよい**条件 (すべて満たす):

- 期待挙動の解釈が一意に定まる
- 適用可能な類似パターンが見つかった
- 公開境界に触れない(HTTP API、proto スキーマ、DB スキーマ、イベント、export された型/シンボル)
- 副作用が局所的(共通ユーティリティ、認可ロジック、横断的関心事に触れない)

**プラン提示すべき**条件 (いずれか該当):

- 受け入れ条件に複数の妥当な解釈がある
- 実装に設計上の選択肢がある (既存拡張 vs 新規追加 / どの層に置くか / どのパターンを採用するか)
- 公開境界に触れる
- 参考パターンが見つからず、設計判断が必要
- 横断的または副作用の影響が読みづらい

プランは簡潔に:

1. 意図の言い直し (1〜2 行)
2. 変更するファイルと、その理由を 1 行ずつ
3. アプローチ — キーとなる設計判断を簡潔に
4. 仮定 (受け入れ条件が曖昧な場合)
5. スコープ外 (1 行、スコープクリープ防止)

明示的な承認を得てから実行する。

## フェーズ 4: 実装ルール

言語によらず適用:

- **真似る、創るな。** 最も近い既存実装のファイル構造・命名・型・イディオムに合わせる。逸脱する場合は理由を明記。
- **最小差分。** 受け入れ条件が要求するものだけ変更。ついで整形、ついでリファクタ、関係ない依存更新はしない。
- **新しい抽象を導入しない。** タスクが明示的に求めていない限り、「念のため」のレイヤー追加は悪臭。
- **公開境界は不変。** それを変えること自体がタスクでない限り触らない。
- **型キャストでなく型修正。** `as` / `!!` / `any` / `unchecked` を避け、根本の型を直す。

言語別のルール:

- Kotlin → `references/kotlin.md`
- React TypeScript → `references/react-ts.md` (UI 再現は `frontend-fidelity` に委譲)
- protobuf → `references/protobuf.md`
- Node.js → `references/nodejs.md`

該当言語のコードを書く前に、対応する reference を読むこと。

## UI 委譲プロトコル

UI 実装は **同一コンテキスト内で `Skill` tool を使って `frontend-fidelity` を起動**して委譲する。subagent (Agent tool) ではなく Skill として呼ぶことで、dev-task がフェーズ 1〜3 で集めた成果物をそのまま渡せる。

### 委譲時に渡す情報

`args` または直前のメッセージに以下を構造化して含める:

- **意図** — タスクを 1〜2 行で言い直したもの (フェーズ 1 の成果)
- **受け入れ条件** — Linear / 自然言語から抽出した条件のリスト
- **仮定** — 曖昧な箇所について dev-task が置いた仮定
- **Figma ソース** — フレーム URL / ノード ID (チケットや会話にあれば。なければ "未指定" と書く)
- **対象 story / コンポーネント** — フェーズ 2 で特定した既存ファイルパス (拡張対象 or 新規追加先)
- **類似実装** — フェーズ 2 で見つけた参照パターンのパス
- **スコープ外** — プラン提示時に決めた除外項目

これらが揃っていれば `frontend-fidelity` はフェーズ 1〜3 (ソース確定・棚卸し) を短縮できる。Figma URL のみ欠落している場合、`frontend-fidelity` 側がユーザーに要求する。

### 委譲後の責務と検証順序

`frontend-fidelity` は **視覚チェック反復が収束した時点 (フェーズ 8 内、ユーザー最終確認の前)** で制御を返してくる。dev-task はそこから:

1. **フェーズ 5 (型/ビルド/lint) を実行** — `frontend-fidelity` は視覚比較が中心で `tsc --noEmit` / `npm run lint` を含まないため必須
2. **型エラー等で実装を修正した場合は `frontend-fidelity` を再起動** — 視覚チェック反復からやり直す (構造が変わったため)
3. **フェーズ 5 通過後、ユーザー最終確認を提示** — `<project-root>/visual-check/` に残っている Playwright スクショと、Figma フレーム画像を side-by-side で見せて承認を得る

順序を「視覚反復 → 型/ビルド/lint → ユーザー最終確認」にすることで、型エラー修正のたびに最終確認をやり直す手戻りを避ける。

## フェーズ 5: 検証

プロジェクトが対応する範囲で、以下の順に実行:

1. **型チェック / コンパイル** — `./gradlew compileKotlin`、`tsc --noEmit`、`npm run typecheck`、`buf lint` / `buf breaking`
2. **ビルド** — `./gradlew build`、`npm run build`
3. **関連テスト** — 変更箇所をカバーするテストのみ。変更範囲が広い、またはユーザーが要求した場合のみ全件実行。
4. **lint / フォーマット** — プロジェクト定義のコマンド (`./gradlew ktlintCheck`、`npm run lint` 等)

失敗したら修正して再実行。最終結果をユーザーに報告。

UI 委譲があった場合は、ここを通過してから「UI 委譲プロトコル > 委譲後の責務と検証順序」に従い視覚最終確認を提示する。

完了は「ローカルチェック通過」(+ UI 委譲があればユーザー視覚最終確認)まで。コミット / push / PR は明示的に依頼されない限りユーザー側の操作。
