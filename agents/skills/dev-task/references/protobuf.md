# protobuf

`.proto` を変更する前に読むこと。

> スキーマ変更は公開境界に触れる。`dev-task` フェーズ 3 のルールにより、原則プラン提示が必要。

## 後方/前方互換性

これらは破ると本番のクライアントが壊れる絶対ルール。

- **フィールド番号を絶対に変えない。** フィールド番号はワイヤーフォーマット。番号を振り直すと旧スキーマで動くクライアントのデータが静かに壊れる。
- **削除したフィールド番号を再利用しない。** 削除時は番号を `reserved` で予約:
  ```proto
  reserved 4, 7, 12 to 14;
  reserved "old_field_name";
  ```
- **フィールド型を非互換に変えない。** 一部はワイヤー互換 (`int32` ↔ `int64` ↔ `uint32` ↔ `uint64` ↔ `bool`、ただし注意あり)。多くは非互換。迷うなら新規フィールドを追加し旧フィールドを deprecated にする。
- **proto2 で `required` ↔ `optional` を変更しない。** proto3 でも `optional` の有無で presence が変わるため、既存フィールドの状況を確認してから合わせる。

## フィールド追加

- 新規 optional フィールドの追加は安全。旧クライアントは無視、新クライアントは旧メッセージからデフォルト値を読む。
- 空いている次の番号を取る。`reserved` を確認せずに穴埋めしない。
- 高頻度フィールドは番号 1〜15 を使う(1 バイトエンコード)。
- 意味をコメントで残す。proto は契約そのもの。

## フィールド削除

- 同じ変更で `reserved <番号>` と `reserved "<名前>"` を入れる。
- 利用者と調整する。削除は多段階で進める: deprecated 化 → 利用停止 → 削除。

## enum

- 0 値は必ず `…_UNSPECIFIED`。常に。
- enum 値の並び替え禁止。整数値がワイヤーフォーマット。
- enum 値の追加は安全。旧クライアントは「未知の値」として扱う(言語のライブラリ実装に依存)。

## サービス / RPC 変更

- メソッド追加: 安全。
- メソッド削除/リネーム: 破壊的。
- リクエスト/レスポンス型の変更: 破壊的。
- `stream` キーワード(ストリーム種別)は契約の一部。

## 生成コード

- 生成ファイルを直接編集しない。生成結果が間違っているなら proto か codegen 設定が間違っている。
- `buf` が設定されていれば `buf breaking` で検証。なければ本チェックリストで diff を手で確認。

## スタイル

- ファイル先頭の順序: `syntax`、`package`、`option go_package` / `option java_package`、import、型定義。
- メッセージ名は PascalCase、フィールド名は snake_case。Google のスタイルガイドに従う。
- 1 つの概念につき 1 つのトップレベルメッセージ。ネストメッセージは親の文脈でのみ意味を持つ場合のみ。
